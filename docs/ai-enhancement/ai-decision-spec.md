# AI 判断规范

## 文档目标

这份文档用于定义第一阶段 AI 判断能力的核心规则，避免开发过程中边做边改、判断标准漂移。

第一阶段 AI 的定位不是完整评分器，而是现有规则筛选之后的二次判断器。

## 第一阶段判断目标

AI 需要回答的核心问题是：

`这个 JD 是否值得投？`

第一阶段 AI 输出优先是简短、可执行、低成本的结论，而不是复杂长分析。

## 第一阶段判断类别

### 1. 值得投

适用场景：

- 岗位主方向与简历主方向高度一致
- 核心技术栈有较高重合
- 不存在明显硬性门槛缺失
- 适合占用当天投递名额

理解方式：

`整体方向对，主要能力匹配，建议直接投递。`

### 2. 可尝试

适用场景：

- 岗位和简历有一定匹配度
- 存在一些技能缺口或方向偏差
- 不是最优岗位，但仍有投递价值

理解方式：

`不是最理想，但仍值得尝试。`

### 3. 不建议投

适用场景：

- 岗位核心方向和简历差异较大
- 核心技术栈差距过大
- 存在明显硬性门槛不满足
- 即使标题看似接近，JD 实际要求也不匹配

理解方式：

`投递命中概率低，不值得占用投递名额。`

## AI 判断时重点考虑的因素

第一阶段建议 AI 重点关注以下因素：

### 1. 岗位方向是否一致

例如：

- 全栈 vs 全栈：方向可能一致
- 全栈 vs Java 企业级后端：方向可能部分一致
- 全栈 vs 运维 / 测试 / 产品：方向可能不一致

### 2. 核心技术栈是否匹配

例如：

- 简历偏 Python / Vue / PHP / RAG
- JD 偏 Java / Spring Boot / Spring Cloud / MyBatis

此时应识别为存在明显技术栈偏差。

### 3. 是否存在硬性门槛

例如：

- 必须若干年某主技术栈项目经验
- 必须某类学历
- 必须某类行业背景

### 4. 技能缺口是否可接受

例如：

- 缺 Redis / Docker 这类次级技能，通常还能尝试
- 缺整个主技术栈，通常不建议投

## 第一阶段推荐的判断流程

### 第一步：规则先筛选

先由现有非 AI 规则筛选处理：

- 城市
- 薪资
- 经验
- 公司黑白名单
- 岗位关键词
- JD 正则

### 第二步：AI 做二次判断

AI 对通过规则筛选的岗位进行“值得投 / 可尝试 / 不建议投”的判断。

### 第三步：程序执行结果

- `值得投`：进入正常投递流程
- `可尝试`：可继续投递或排在后面
- `不建议投`：直接跳过

## JSON 输出协议

第一阶段建议统一输出如下结构：

```json
{
  "decision": "recommend",
  "summary": "岗位整体与你的简历方向匹配，建议投递。",
  "matchedSkills": ["Vue", "JavaScript"],
  "missingSkills": ["Spring Boot", "Docker"]
}
```

## 字段说明

### `decision`

允许值：

- `recommend`
- `try`
- `reject`

### `summary`

要求：

- 一句话
- 简短
- 说明主要原因
- 不要写成长段解释

### `matchedSkills`

要求：

- 可选字段
- 仅返回核心已匹配技能
- 优先返回真实技术项

### `missingSkills`

要求：

- 可选字段
- 仅返回关键缺口技能
- 尽量避免“沟通能力”“学习能力”这类泛化项

## decision 映射规则

### recommend -> 值得投

满足以下倾向：

- 岗位方向匹配较高
- 技术栈匹配较高或中等
- 不存在明显硬门槛

### try -> 可尝试

满足以下倾向：

- 岗位方向有一定相关性
- 技术栈存在缺口但不是完全不相关
- 仍有尝试价值

### reject -> 不建议投

满足以下倾向：

- 岗位方向匹配低
- 核心技术栈差异过大
- 存在明显硬门槛

## 第一阶段缓存策略

### 默认策略

- 同一岗位优先复用历史 AI 判断结果
- 只有未分析过的岗位才调用 LLM

### 后续扩展

以后可以再加入：

- 简历版本号
- 模型版本号
- 手动重跑机制

## 第一阶段边界

第一阶段不强做这些内容：

- 精确 0-100 匹配分
- 全量岗位排序
- 复杂 150 个岗位补足策略
- 历史投递复盘
- RAG 知识库检索
- LangChain 工作流编排

## 第一阶段成功标准

- AI 能稳定输出结构化结果
- 用户能快速看到“值不值得投”
- 对值得投岗位能看到技能缺口
- 自动开聊能正确跳过“不建议投”的岗位
- 整体成本可控，不依赖全量深度分析
