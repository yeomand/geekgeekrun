# 自动开聊 AI 增强 TODO

## 当前开发策略

第一阶段先收敛成一个真正可落地、成本可控、适合写进 GitHub 和简历的版本，只做这 3 件事：

- [ ] AI 简短投递判断
- [ ] 针对“值得投”岗位的技能缺口分析
- [ ] 自动开聊中跳过“不建议投”的岗位

后续更复杂的 AI 排序、150 份投递额度优化、历史投递复盘等能力，先保留在本文档后半部分，等第一阶段稳定后再扩展。

## 第一阶段目标

第一阶段不追求“对所有岗位做完整 AI 深度分析”，而是做一套低成本、可解释、能接入现有自动开聊流程的 AI 二次判断机制。

- [ ] 继续复用现有非 AI 规则筛选，先过滤明显不合适的岗位
- [ ] AI 只负责判断“这个 JD 是否值得投”
- [ ] 第一版 AI 优先输出简短结论，而不是复杂长分析
- [ ] AI 输出建议形态优先采用：`值得投 / 可尝试 / 不建议投`
- [ ] 只返回一句简短原因，说明为什么值得投或不值得投
- [ ] 技能缺口分析只针对“值得投”或高优先级岗位执行
- [ ] 候选岗位采用分批分析，而不是一次性分析全部岗位
- [ ] 分析结果本地缓存，避免重复消耗 token
- [ ] 目标是在控制成本的前提下，提高自动开聊的岗位匹配质量

## 第一阶段难点与风险

### 1. AI 输出稳定性

- [ ] 大模型可能不按约定返回结构化 JSON
- [ ] `decision` 字段可能不稳定，导致前端和流程判断困难
- [ ] `summary` 可能过长、跑题或表达不统一
- [ ] 需要通过 prompt 约束和解析兜底保证结果可用

### 2. 简历摘要与 JD 裁剪

- [ ] 发给 AI 的内容太多会增加 token 成本
- [ ] 发给 AI 的内容太少会导致判断不准
- [ ] 需要设计低成本但信息足够的简历摘要格式
- [ ] 需要确定 JD 中哪些字段必须参与分析

### 3. 自动开聊主流程接入

- [ ] AI 判断接入后可能拖慢现有自动开聊流程
- [ ] AI 出错时必须能安全回退到原有规则流程
- [ ] 需要避免因为 AI 超时或失败影响现有可用功能
- [ ] 需要保证开启 AI 和关闭 AI 两种模式都稳定

### 4. 本地缓存策略

- [ ] 需要明确同一岗位是否直接复用历史 AI 结果
- [ ] 需要考虑简历更新后历史结果是否失效
- [ ] 需要支持手动重新分析单个岗位
- [ ] 缓存策略如果不清晰，容易导致结果误用

### 5. 技能缺口分析的有效性

- [ ] AI 可能给出过于宽泛的“软技能”缺口
- [ ] 技能缺口需要尽量偏真实技术项，而不是空泛结论
- [ ] 需要限制技能缺口分析只对值得投岗位执行，避免做重

### 6. 规则与 AI 的边界

- [ ] 需要明确哪些问题继续交给规则判断
- [ ] 需要明确哪些问题交给 AI 做二次判断
- [ ] 避免规则和 AI 互相冲突，导致结果不可解释

### 7. UI 展示的克制性

- [ ] 第一阶段结果不应展示得过重，避免界面复杂化
- [ ] 需要让用户一眼看懂“值不值得投”和“差什么技能”
- [ ] 需要避免 AI 信息干扰现有操作节奏
